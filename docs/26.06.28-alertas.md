# Resumo sobre alertas de segurança em projetos TypeScript/JavaScript

Resumidamente: é comum ter muitos alertas em um projeto TypeScript/JavaScript porque o ecossistema tem muitas dependências (e dependências transitivas) e o Dependabot/GitHub Security gera um alerta para cada CVE encontrado na árvore de dependências — portanto poucas bibliotecas vulneráveis ou um lockfile desatualizado já geram muitos alertas.

## Alertas de segurança: causas e soluções

O que observei (com base nos dados que você enviou: repo jocile/estudos-biblicos, ~65% TypeScript + muitas dependências front-end):

- Muitos pacotes: projetos JS/TS normalmente dependem de muitas bibliotecas e ferramentas de build/test — cada uma pode trazer vulnerabilidades transitivas.
- Lockfile desatualizado: se não for atualizado com frequência, alertas acumulam porque versões corrigidas não estão instaladas.
- Dependências de desenvolvimento também disparam alertas (linters, bundlers, test runners).
- Versões major antigas: atualizações incompatíveis costumam ser evitadas e isso mantém versões vulneráveis no projeto.
- Dependências transientes: você pode não usar diretamente a biblioteca vulnerável, mas ela entra via outra dependência.
- Configuração do Dependabot/segurança: se estiver habilitado para tudo, vai listar todos os CVEs — pode parecer “muitos” mesmo para riscos baixos.

O que fazer (passo a passo curto):

1. Ver diagnóstico local
   - npm: npm audit && npm outdated
   - yarn: yarn audit && yarn outdated
   - pnpm: pnpm audit && pnpm outdated
2. Tentar correções automáticas seguras
   - npm audit fix
   - se necessário e com cuidado: npm audit fix --force (pode aplicar quebras)
   - yarn upgrade-interactive para escolher atualizações
3. Atualizar lockfile e instalar clean
   - rm -rf node_modules package-lock.json && npm install (ou yarn.lock/pnpm-lock)
4. Priorizar por risco
   - Na aba Security → Dependabot do GitHub, filtre por severity/critical e por pacote usado em runtime.
   - Trate primeiro vulnerabilidades que afetam produção (runtime) e com severidade alta/critica.
5. Reduzir a superfície
   - Remover dependências não usadas.
   - Substituir bibliotecas historicamente problemáticas por alternativas menores/mais mantidas.
6. Automatizar
   - Habilitar Dependabot security updates (abre PRs automaticamente).
   - Configurar dependabot.yml para atualizar com frequência adequada e ignorar atualizações irrelevantes.
7. Testes & CI
   - Rodar testes e checagens em PRs de atualização para evitar regressões.
8. When stuck: considerar ferramentas externas (Snyk, GitHub Advanced Security) para sugestões de remediação e backports.

## Exemplo de dependabot.yml

Exemplo mínimo de dependabot.yml (se quiser que eu crie):

- atualizações semanais para npm:

  ```yml
  version: 2
  updates:
    - package-ecosystem: "npm"
      directory: "/"
      schedule:
        interval: "weekly"
      allow:
        - dependency-type: "direct"
  ```

## Configuração do Dependabot

Aqui está uma proposta de .github/dependabot.yml que abre PRs automáticos para atualizar dependências (npm + GitHub Actions), com atualizações semanais e limites para evitar excesso de PRs:

```yaml name=.github/dependabot.yml
version: 2

updates:
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "weekly"
      day: "monday"
    open-pull-requests-limit: 5
    allow:
      - dependency-type: "all"
    labels:
      - "dependencies"
      - "dependabot"
    commit-message:
      prefix: "chore(deps)"
    ignore:
      # Ignora atualizações major (para reduzir quebras inesperadas).
      - dependency-name: "*"
        update-types:
          - "version-update:semver-major"

  - package-ecosystem: "github-actions"
    directory: "/"
    schedule:
      interval: "weekly"
    open-pull-requests-limit: 5
    labels:
      - "dependencies"
```

Observações rápidas:

- Dependabot já abrirá PRs automaticamente com essa configuração; você pode revisar e mesclar as PRs manualmente.
- Ignorei atualizações semver-major por padrão para reduzir risco de quebras. Se preferir que Dependabot também proponha major updates, remova a seção ignore.
- Ajustes recomendados conforme preferência:
  - Intervalo: daily / weekly / monthly.
  - open-pull-requests-limit: aumentar se quiser mais PRs simultâneas.
  - Adicionar reviewers/assignees ou automerge (automerge exige configuração adicional/ações).
- Se quiser, eu posso:
  - Gerar uma versão sem ignorar major updates.
  - Incluir automerge para PRs com segurança/CI aprovados.
  - Criar o arquivo no repositório e abrir PR automaticamente (preciso da sua confirmação para realizar a escrita).
